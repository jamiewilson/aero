## The problem

Build-time `{ }` interpolation updates HTML once at compile time. For counters, toggles, form state, or conditional UI that changes after load, you either add Alpine.js or HTMX, or write imperative DOM code. Both work, but they are separate libraries with their own attribute syntax and bootstrapping.

## What this looks like without Aero

```html
<button id="inc">+</button>
<span id="count">0</span>
<script>
	let count = 0
	document.getElementById('inc').addEventListener('click', () => {
		count++
		document.getElementById('count').textContent = String(count)
	})
</script>
```

Every binding is manual: find elements, wire listeners, keep DOM in sync.

## How Aero helps

Aero ships an **optional** reactivity layer. Enable `reactivity: true` in config, declare state in `<script is:state>`, and use bindings (`{ count }`, `on:click`, `show`, `value`, reactive `if` / `for`) directly in markup. The compiler emits a small client mount; there is no Virtual DOM and no mandatory framework bundle for static pages.

Reactivity is **off by default**. Static sites stay static until you opt in.

<Tip>
	HTMX and Alpine.js still work unchanged. Use Aero reactivity when you want state and bindings in the same template syntax as your build-time markup.
</Tip>

## Enable reactivity

In `aero.config.ts`:

```ts aero.config.ts
import { defineConfig } from '@aero-js/config'

export default defineConfig({
	reactivity: true,
})
```

Or pass the flag to the Vite plugin:

```ts vite.config.ts
import { aero } from '@aero-js/vite'

export default defineConfig({
	plugins: [aero({ reactivity: true })],
})
```

`aero check` reports templates that use reactive directives without the flag enabled.

## `<script is:state>`

State scripts run in the browser after the page mounts. Declare bindings with `let`:

```html
<script is:state>
	let count = 0
</script>

<section>
	<button on:click="{ count-- }">−</button>
	<span>{ count }</span>
	<button on:click="{ count++ }">+</button>
</section>
```

Rules:

- One `<script is:state>` per template file.
- Top-level `let` declarations become reactive signals.
- Functions declared at the top level are available to event handlers and bindings.
- `import` statements are hoisted into the mount scope (same as client scripts).

Use `const` for derived helpers that read state but are not themselves signals:

```html
<script is:state>
	let email = ''
	let agree = false

	const isComplete = () => email.length > 0 && agree
</script>

<form class:is-valid="{ isComplete() }">…</form>
```

<Note>
	`<script is:state>` requires `reactivity: true`. Build scripts (`is:build`) and state scripts serve different roles — do not merge them.
</Note>

## Hydration

On first paint, the server-rendered HTML is the source of truth. At mount, Aero reads initial values from a hydration payload and wires bindings. You do not call a hydration API manually; declaring `let` bindings and using them in markup is enough.

## Runtime access

The client entry (`@aero-js/core`) exposes the active runtime when reactivity is enabled:

```ts
import aero from '@aero-js/core'

const runtime = aero.getReactivityRuntime?.()
runtime?.store.get('count') // signal accessor
```

Use this for imperative code (for example adopting HTML inserted after load). See [Adopt runtime fragments](/guide/reactivity/adopt).
