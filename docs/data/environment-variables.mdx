title: "Environment Variables"
description: "Use .env files to manage build-time and client-exposed variables in Aero."
---

Aero uses Vite's environment variable system. Variables are available in build-time code by default; you opt specific ones into the client bundle by prefixing them with `VITE_`.

## Client vs. build-only variables

The prefix on a variable name controls where it's available:

| Prefix   | Where it's available  | Example use                        |
| -------- | --------------------- | ---------------------------------- |
| (none)   | Build and server only | API keys, internal URLs            |
| `VITE_`  | Build and client      | Public API base URL, feature flags |

Variables without a prefix are never included in the browser bundle — they're only accessible in `<script is:build>` and other code that runs at build time. Prefix a variable with `VITE_` to make it available in both build scripts and client `<script>` blocks.

```bash
# .env

# Build/server only — never sent to the browser
API_SECRET=xxx
API_BASE=https://api.internal.example.com

# Client-safe — Vite includes these in the bundle
VITE_PUBLIC_API=https://api.example.com
VITE_FEATURE_X=enabled
```

## Using variables in code

Access variables through `import.meta.env` in both build and client code:

<CodeGroup>
```html Build script (is:build)
<script is:build>
  // Both prefixed and unprefixed vars are available here
  const secret = import.meta.env.API_SECRET
  const publicApi = import.meta.env.VITE_PUBLIC_API
</script>
```

```html Client script
<script>
  // Only VITE_-prefixed vars (and built-ins) are replaced here
  const api = import.meta.env.VITE_PUBLIC_API
  const isDev = import.meta.env.DEV
</script>
```
</CodeGroup>

## .env file loading order

Vite loads env files from the project root in this order (later files override earlier ones):

1. `.env` — all modes
2. `.env.local` — all modes, typically git-ignored
3. `.env.[mode]` — e.g. `.env.development`, `.env.production`
4. `.env.[mode].local` — mode-specific, typically git-ignored

Shell environment variables win over `.env` files. Restart the dev server after changing any `.env` file.

<Tip>
  You can change the client-exposed prefix with Vite's [`envPrefix`](https://vite.dev/config/shared-options.html#envprefix) option. For example, set `envPrefix: 'PUBLIC_'` to use `PUBLIC_` instead of `VITE_`.
</Tip>

## Built-in Vite variables

Vite provides these out of the box — no `.env` entry needed:

| Variable                  | Value                                          |
| ------------------------- | ---------------------------------------------- |
| `import.meta.env.MODE`    | `'development'` or `'production'`              |
| `import.meta.env.DEV`     | `true` in dev                                  |
| `import.meta.env.PROD`    | `true` in production build                     |
| `import.meta.env.BASE_URL`| Base URL from Vite config (e.g. `'./'`)        |
| `import.meta.env.SSR`     | `true` when running in SSR / build             |

## Aero-injected variable

When you set `site.url` in your Aero config, the plugin automatically injects:

| Variable                  | Value                                          |
| ------------------------- | ---------------------------------------------- |
| `import.meta.env.SITE`    | Your canonical site URL, e.g. `'https://example.com'` |

You don't need to add `SITE` to `.env` — Aero handles it. See [Site URL](/data/site-url) for configuration details.

## Aero build-time Node variables

These environment variables control Aero's build behavior at the Node/process level. They are **not** available as `import.meta.env` unless you wire them yourself:

| Variable                              | Purpose                                                                                                      |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `AERO_INCREMENTAL`                    | Set to `1`, `true`, or `yes` to enable incremental static prerender, manifest caching, and collection cache. |
| `AERO_LOG`                            | Set to `debug` to log static prerender decisions and timings.                                                |
| `AERO_STATIC_PRERENDER_CONCURRENCY`   | Integer — limits parallel static HTML prerender jobs.                                                        |

```bash
# Run a build with incremental mode and debug logging
AERO_INCREMENTAL=1 AERO_LOG=debug pnpm build
```

## TypeScript declarations

For type-checking and autocomplete on `import.meta.env`, add a declaration file to your project:

```ts
// env.d.ts
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly SITE: string
  // Add your own variables here:
  readonly VITE_PUBLIC_API: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}
```

Include this file in your `tsconfig.json` (or place it in a directory already covered by `include`). The `/// <reference types="vite/client" />` line pulls in Vite's built-in types; the interface merges your custom variables on top.
