Aero supports TypeScript in `<script is:build lang="ts">` blocks, content files, and shared type modules. The VS Code extension and language server provide IntelliSense and diagnostics for build scripts automatically — you get autocomplete and type errors in your HTML templates without any extra setup.

## Typing component props

`Aero.props` is typed as `Record<string, any>` by default. To add type safety, define an interface and cast when you destructure:

```html
<!-- client/components/greeting.html -->
<script is:build lang="ts">
  interface GreetingProps {
    name: string
    title?: string
  }

  const { name, title = 'Guest' } = Aero.props as GreetingProps
</script>

<h1>Hello, { name }!</h1>
```

### Shared prop types

For types shared across multiple components, define them in a `.ts` file and import them:

```ts
// client/types/props.ts
export interface HeaderProps {
  title: string
  subtitle?: string
}
```

```html
<!-- client/components/header.html -->
<script is:build lang="ts">
  import type { HeaderProps } from 'types/props'

  const { title, subtitle } = Aero.props as HeaderProps
</script>

<header>
  <h1>{ title }</h1>
</header>
```

<Tip>
  Avoid the `@types/*` prefix for your own type modules — it is reserved for DefinitelyTyped packages. Use a path like `types/props` or add a custom alias such as `@project-types/*` in `tsconfig.json`.
</Tip>

### Typed props when passing

Type the object you spread so TypeScript catches missing required props at the call site:

```html
<!-- client/pages/index.html -->
<script is:build lang="ts">
  import header from '@components/header'
  import site from '@content/site'
  import type { HeaderProps } from 'types/props'

  const headerProps: HeaderProps = {
    title: site.home.title,
    subtitle: site.home.subtitle,
  }
</script>

<header-component props="{ ...headerProps }" />
```

## Typing content collections

Use a [Standard Schema](https://standardschema.dev)-compliant validator (Zod, ArkType, Valibot) in `content.config.ts` to type your collection entries:

```ts
// content.config.ts
import { defineConfig, defineCollection } from '@aero-js/content'
import { z } from 'zod'

const docsSchema = z.object({
  title: z.string(),
  subtitle: z.string().optional(),
  published: z.number(),
})

export default defineConfig({
  collections: [
    defineCollection({
      name: 'docs',
      directory: 'client/content/docs',
      schema: docsSchema,
    }),
  ],
})
```

### Typed getStaticPaths

Use the schema-inferred types in dynamic page scripts:

```html
<!-- client/pages/docs/[slug].html -->
<script is:build lang="ts">
  import { getCollection, render } from 'aero:content'

  export async function getStaticPaths() {
    const docs = await getCollection('docs')
    return docs.map(doc => ({
      params: { slug: doc.id },
      props: doc,
    }))
  }

  const doc = Aero.props
  const { html } = await render(doc)
</script>

<h1>{ doc.data.title }</h1>
<article>{ html }</article>
```

For stricter typing on `doc.data`, add an inline interface and assert:

```html
<script is:build lang="ts">
  import { getCollection, render } from 'aero:content'

  interface DocData {
    title: string
    subtitle?: string
    published: number
  }

  export async function getStaticPaths() {
    const docs = await getCollection('docs')
    return docs.map(doc => ({
      params: { slug: doc.id },
      props: doc,
    }))
  }

  const doc = Aero.props
  const data = doc.data as DocData
  const { html } = await render(doc)
</script>

<h1>{ data.title }</h1>
```

## Ambient types and env.d.ts

Aero ships ambient declarations in `@aero-js/core/env` that make `Aero`, `renderComponent`, and `*.html` imports available to TypeScript. Include them in one of two ways:

<CodeGroup>

```json tsconfig.json (types array)
{
  "compilerOptions": {
    "types": ["@aero-js/core/env"]
  }
}
```

```ts triple-slash reference
/// <reference types="@aero-js/core/env" />
```

</CodeGroup>

<Note>
  For `.html` build scripts, the VS Code extension and language server inject these declarations automatically. You only need the above for `.ts` files outside of templates.
</Note>

### Typing import.meta.env

Create an `env.d.ts` in your project root to add types for `import.meta.env.SITE` and any custom `VITE_` variables:

```ts
// env.d.ts
/// <reference types="@aero-js/core/env" />

interface ImportMetaEnv {
  readonly SITE: string
  readonly VITE_API_URL: string
  // add more VITE_ variables here
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}
```

Make sure `env.d.ts` is included in your `tsconfig.json` via `include` or `files`.

## Path alias types

For path aliases to resolve correctly in the TypeScript language server, keep `tsconfig.json` paths in sync with the aliases Aero injects at build time. See [Path Aliases](/configuration/path-aliases) for the full default alias table and how to add custom aliases.

## CI type-checking

The language server type-checks your build scripts in the editor. To enforce the same checks in CI without an IDE, run:

```bash
pnpm exec aero check --types
```

This runs compile checks first, then TypeScript on build scripts and `{ }` interpolation sites using your workspace `tsconfig.json`. See [CLI](/tooling/cli) for flags and exit codes.
