When you run `pnpm create @aero-js my-app`, Aero scaffolds this structure:

```plaintext
my-app/
├── client/
│   ├── assets/
│   │   ├── scripts/        # Client-side TypeScript/JavaScript
│   │   └── styles/         # CSS files
│   ├── components/         # Reusable .html components
│   ├── layouts/            # Layout wrappers with <slot>
│   └── pages/              # File-based routing
├── content/
│   └── site.ts             # Global site data
├── public/                 # Static assets (copied as-is)
├── vite.config.ts          # Aero Vite plugin
└── tsconfig.json           # Path aliases
```

The fullstack template adds:

```plaintext
my-app/
├── server/
│   └── api/                # Nitro API route handlers
└── nitro.config.ts         # Nitro configuration
```

## Directories

### `client/pages/`

Every `.html` file here becomes a route. The file path maps directly to the URL:

| File                            | URL           |
| ------------------------------- | ------------- |
| `client/pages/index.html`       | `/`           |
| `client/pages/about.html`       | `/about`      |
| `client/pages/docs/index.html`  | `/docs`       |
| `client/pages/blog/[slug].html` | `/blog/:slug` |

A page can be as simple as plain HTML, or it can include a `<script is:build>` block to import layouts, components, and data:

```html
<script is:build>
	import base from '@layouts/base'
	import header from '@components/header'
	import site from '@content/site'
</script>

<base-layout>
	<header-component title="{ site.title }" />
	<p>{ site.description }</p>
</base-layout>
```

### `client/layouts/`

Layouts are `.html` files that wrap pages. They expose a `<slot />` where page content is inserted. Import a layout in your page's `<script is:build>` and use it as `<name-layout>`:

```html
<!-- client/layouts/base.html -->
<html lang="en">
	<head>
		...
	</head>
	<body>
		<slot />
	</body>
</html>
```

The file name determines the tag name: `base.html` → `<base-layout>`.

### `client/components/`

Components are reusable `.html` files. Import them in a page or layout and use them as `<name-component>`. Pass data via attributes or the `props` attribute; read it with `Aero.props`:

```html
<!-- client/components/greeting.html -->
<script is:build>
	const { name } = Aero.props
</script>

<h2>Hello, { name }!</h2>
```

The file name determines the tag name: `greeting.html` → `<greeting-component>`.

### `client/assets/`

CSS and client-side scripts live here, organized under `styles/` and `scripts/`. Reference them with path aliases:

```html
<link rel="stylesheet" href="@styles/global.css" />
```

```html
<script is:build>
	import someFunction from '@scripts/someModule'
</script>
```

Vite handles bundling, autoprefixing, and minification automatically.

### `content/`

Put TypeScript or JavaScript data modules here. `content/site.ts` is created by the scaffold and is a good place for global site data:

```ts
// content/site.ts
export default {
	meta: {
		title: 'My Site',
		description: 'Built with Aero',
	},
	nav: [
		{ label: 'Home', path: '/' },
		{ label: 'About', path: '/about' },
	],
}
```

Import it anywhere using the `@content/` alias:

```html
<script is:build>
	import site from '@content/site'
</script>

<title>{ site.meta.title }</title>
```

For Markdown-driven content (blog posts, docs), use `getCollection()` and `render()` from `aero:content`. See [Content Collections](/data/content-collections).

### `public/`

Files in `public/` are copied to `dist/` as-is without processing. Use it for favicons, `robots.txt`, open graph images, and other static assets that don't need Vite's pipeline.

### `server/` (fullstack template only)

API route handlers live under `server/api/`. Each file maps to a URL under `/api/` by default. File names can include HTTP method suffixes:

```ts
// server/api/submit.post.ts  →  POST /api/submit
import { defineHandler, readBody } from 'nitro/h3'

export default defineHandler(async event => {
	const body = await readBody(event)
	return { ok: true, message: body.message }
})
```

This directory is powered by Nitro. It only exists when you use the fullstack template or set `server: true` in your config.

## Config files

### `vite.config.ts`

The entry point for Aero's Vite plugin. At minimum it registers the `aero()` plugin:

```ts
import { defineConfig } from 'vite'
import { aero } from '@aero-js/vite'

export default defineConfig({
	plugins: [aero()],
})
```

Pass options directly to `aero()` to configure the site URL, redirects, content collections, or Nitro:

```ts
plugins: [
	aero({
		site: { url: 'https://example.com' },
		content: true,
		server: true,
	}),
]
```

### `aero.config.ts`

Projects using `@aero-js/config` (included in the fullstack template) can put Aero options in a dedicated `aero.config.ts`. This keeps `vite.config.ts` minimal and gives you typed, env-aware config:

```ts
// aero.config.ts
import { defineConfig } from '@aero-js/config'

export default defineConfig({
	site: { url: 'https://example.com' },
	content: true,
	server: true,
})
```

```ts
// vite.config.ts
import { createViteConfig } from '@aero-js/config/vite'
import aeroConfig from './aero.config.ts'

export default createViteConfig(aeroConfig)
```

See [Configuration](/configuration/aero-config) for all available options.

### `tsconfig.json`

Provides TypeScript compiler options and path aliases for your editor and `tsc`. The scaffold sets up aliases for all standard directories:

| Alias           | Resolves to               |
| --------------- | ------------------------- |
| `@components/*` | `client/components/*`     |
| `@layouts/*`    | `client/layouts/*`        |
| `@pages/*`      | `client/pages/*`          |
| `@content/*`    | `content/*`               |
| `@styles/*`     | `client/assets/styles/*`  |
| `@scripts/*`    | `client/assets/scripts/*` |

<Note>
	`tsconfig.json` is optional for Aero's build — the framework provides all standard path aliases at
	build time. Add it when you need TypeScript compiler options, IDE type checking, or custom
	aliases.
</Note>

### `nitro.config.ts` (fullstack template only)

Nitro-native configuration: storage, cache, database, tasks, plugins, and deployment presets. Use this file for anything that goes beyond the `server: true` toggle in `aero.config.ts`.

```ts
// nitro.config.ts
import { defineNitroConfig } from 'nitro/config'

export default defineNitroConfig({
	preset: 'cloudflare_pages',
})
```

See [Nitro Server](/server/nitro) for more.
